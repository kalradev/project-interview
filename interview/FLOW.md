# Interview platform – end-to-end flow

## Overview

1. **Resumes** are collected (e.g. from Naukri or manual/CSV) and appear on the **admin dashboard**.
2. Admin **imports a candidate** → system creates a user, generates a **password**, and sends an **email** with:
   - Login credentials (email + password)
   - **Interview time** (exact date/time, or “within 1–2 days”)
   - **Setup link** to download and install the Interview Agent app
3. **Candidate** installs the app on their PC, then **logs in** with that email/password.
4. After login, the **interview screen locks** (fullscreen); the candidate completes the interview (questions by **job role** and **tech stack**).
5. **Photo** can be captured at start (to reduce proxy attempts); **voice/lip sync** checks can be added later.
6. Every **question–answer** and a **summary** are stored and shown in the **admin dashboard** as a **report**.
7. **Admin** can **allow next round**, **select**, or **reject** the candidate.

---

## 1. Resume collection → admin dashboard

- **Naukri:** Today there is no Naukri API in the repo. You can:
  - Use **manual import**: add candidates one-by-one via **POST /api/v1/admin/candidates** (email, full_name, job_role, tech_stack, resume_text, etc.).
  - Or build a **CSV import** (same endpoint with a list) or a **Naukri integration** when you have employer API access.
- Resumes/candidates are stored as **users** + **candidate_profiles**. The “admin dashboard” is the set of **GET /api/v1/admin/candidates** and related admin APIs (list, report, action).

---

## 2. Auto email (credentials + time + setup)

- When you **import a candidate** (**POST /api/v1/admin/candidates**) with `send_email: true`:
  - A **user** is created with a **random password**.
  - A **candidate_profile** is created (job_role, tech_stack, `interview_scheduled_at`).
  - An **email** is sent (if SMTP is configured in `.env`) with:
    - Email and **password**
    - **Interview timing** (from `interview_scheduled_at` or a default “within 1–2 days”)
    - **Setup link** (`SETUP_APP_DOWNLOAD_URL` in `.env` – e.g. link to your Interview Agent installer).

Configure in `.env`:

- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM_EMAIL`
- `SETUP_APP_DOWNLOAD_URL` = URL where candidates download the Interview Agent setup.

---

## 3. Candidate: install app → login → screen lock

- Candidate **downloads** the setup from the link in the email and **installs** the Interview Agent (Electron) app.
- In the app they **log in** with the **email and password** from the email (backend **POST /api/v1/auth/login**).
- After login, the app can call **GET /api/v1/candidate/me** to get profile and **POST /api/v1/candidate/me/session** to get or create a session, then switch to **fullscreen/lock** and start the interview (questions from **POST /api/v1/interview/next-question** with `job_role` / tech stack from profile).

---

## 4. Interview by job role + tech stack

- **Job role** and **tech stack** come from the candidate’s **profile** (set at import).
- The desktop app calls **POST /api/v1/interview/next-question** with:
  - `job_role` (and optionally tech_stack in metadata) so the AI asks role- and tech-relevant questions.
- Each time the candidate submits an answer, the app should call **POST /api/v1/interview/record-exchange** (session_id, question_index, question_text, answer_text) so the **admin report** has every Q&A.

---

## 5. Photo and voice/lip sync

- **Photo:** The desktop app can capture a photo (e.g. via camera) at interview start and upload it; backend can store the URL in `candidate_profiles.photo_url`. (Upload endpoint and storage can be added when you implement it.)
- **Voice/lip sync:** Not implemented yet. You can add later: e.g. record a short clip, run a model to check voice vs face/lips, and store a “suspicious” flag or event.

---

## 6. Report and summary on admin dashboard

- **Per-session data:**
  - **Interview exchanges** are stored via **POST /api/v1/interview/record-exchange**.
  - **Summary** can be set when the interview ends via **PATCH /api/v1/sessions/{session_id}/summary** (e.g. generated by your backend or the app).
- **Admin** fetches the full report with **GET /api/v1/admin/candidates/{candidate_id}/report** (exchanges + summary + integrity score/risk).

---

## 7. Admin: allow next round / select / reject

- **POST /api/v1/admin/candidates/{candidate_id}/action** with body `{ "status": "next_round" | "selected" | "rejected" }`.
- This updates the candidate’s status so the dashboard and any next-round flow can use it.

---

## API summary

| Purpose | Method | Path |
|--------|--------|------|
| Import candidate (send invite email) | POST | /api/v1/admin/candidates |
| List candidates | GET | /api/v1/admin/candidates |
| Get candidate report | GET | /api/v1/admin/candidates/{id}/report |
| Admin action (next_round/select/reject) | POST | /api/v1/admin/candidates/{id}/action |
| Candidate: my profile + session | GET | /api/v1/candidate/me |
| Candidate: get or create session | POST | /api/v1/candidate/me/session |
| Record one Q&A | POST | /api/v1/interview/record-exchange |
| Set session summary | PATCH | /api/v1/sessions/{id}/summary |
| Candidate end own session | POST | /api/v1/sessions/end-my |
